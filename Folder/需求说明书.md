**医院预约挂号系统需求规格说明书**

**一、引言**

**1.1 文档目的**

本文档为医院预约挂号系统的官方需求规格说明书，完整、清晰、无歧义地定义系统的业务边界、功能需求、非功能需求、用户角色、业务规则等核心内容。本文档将作为本项目系统设计、代码开发、测试验证、项目验收全流程的核心依据，同时为项目所有干系人提供统一的需求认知基准，确保最终交付的系统完全契合开题报告中设定的研究目标与业务定位。

**1.2 项目背景**

在“健康中国2030”战略及“互联网+医疗健康”相关政策的双重驱动下，我国医疗服务体系正加速数字化转型。传统线下现场挂号模式普遍存在排队耗时长、号源信息不透明、优质医疗资源分配不均、医患信息流通不畅等核心痛点，既严重降低了患者的就医体验，也制约了医院的运营管理效率。

为解决上述行业痛点，本项目拟开发一套基于**Spring Boot + Vue.js前后端分离架构**的医院预约挂号系统，实现挂号业务全流程线上化，优化医疗资源配置效率，提升患者就医便捷度与医院管理数字化水平。同时，本系统将重点攻克高并发抢号场景下的数据一致性、业务公平性、用户数据安全防护三大核心问题，通过全流程系统化测试保障系统的高可用与高可靠性，本系统仅用于毕业设计与论文撰写，无商用部署需求。

**1.3 文档范围**

本文档覆盖医院预约挂号系统全生命周期的需求定义，核心包含以下内容：系统总体描述、用户角色与权限需求、功能性需求、非功能性需求、接口需求、数据需求、运行环境需求、验收标准。

**1.4 预期读者**

本文档的预期读者包括：项目指导教师、系统前后端开发人员、测试人员、毕业设计评审专家及其他项目相关干系人。

**二、系统总体描述**

**2.1 系统总体定位与建设目标**

本系统是面向中小型医疗机构的轻量化、高可用的线上预约挂号服务平台，仅用于毕业设计与论文撰写，以患者就医全流程便捷化为核心，以医院管理数字化为支撑，打通号源管理、在线预约、订单管控、权限管理全业务链路，替代传统线下挂号模式，解决传统挂号模式的核心痛点。

系统核心建设目标如下：

1.  构建全线上化的预约挂号服务体系，支持用户7×24小时在线完成科室浏览、医生查询、号源预约、订单管理全流程操作，打破线下挂号的时空限制，提升患者就医体验。
2.  搭建标准化的号源与排班管理体系，为管理员提供可视化的排班配置、号源发布、预约记录查询能力，实现号源全生命周期的精细化管控，提升医院运营管理效率。
3.  攻克高并发预约场景的核心技术难点，通过缓存、分布式锁、请求队列等技术方案，保障热门号源释放时段多用户并发抢号场景下的系统稳定性、数据一致性与业务公平性，杜绝号源超卖问题。
4.  构建基础的用户数据安全防护体系，通过身份认证、权限管控等手段，保障用户个人信息的存储与传输安全，符合数据安全相关基础规范。
5.  采用前后端分离的主流技术架构，保证系统的可扩展性、可维护性，为后续功能迭代升级预留技术空间，同时为同类医疗信息化系统的毕业设计提供可复用的开发与测试实践参考。

**2.2 业务总体流程**

本系统的核心业务围绕号源生命周期与预约挂号全流程展开，形成闭环业务链路，总体流程如下：

1.  **号源生成与发布环节**：管理员完成科室、医生基础信息配置后，为医生创建出诊排班计划，设置出诊日期、时段、号源总量等信息，系统自动生成对应号源，管理员完成发布后同步更新至号源池并对外展示。
2.  **用户预约挂号环节**：用户完成账号注册并登录系统后，即可浏览科室与出诊医生信息，查看对应时段号源剩余情况，选择目标号源提交预约申请；系统接收请求后，校验用户预约资格、号源库存状态，通过并发控制机制完成号源锁定与扣减，生成预约订单，同步更新号源状态，完成预约流程。
3.  **订单履约与管理环节**：用户可在个人中心查看预约订单详情，按预约时间前往医院就诊；可随时对未就诊的预约订单发起取消申请，系统接收申请后释放对应号源，更新订单与号源状态；医生可在工作台查看本人出诊时段的预约用户列表，核对预约信息。
4.  **系统管控与运维环节**：管理员可对全系统的科室、医生、用户、号源、订单数据进行统一管理，监控系统运行状态，处理异常业务订单，配置系统基础规则，保障系统稳定运行。

**2.3 系统用户角色概述**

本系统设定三类核心用户角色，不同角色对应不同的业务边界与操作权限，角色定义如下：

1.  **患者用户角色**：系统核心服务对象，为有就医挂号需求的普通用户。核心操作围绕预约挂号全流程展开，完成账号注册后即可使用系统全量预约功能，是系统前端门户的主要使用者。
2.  **医生角色**：医院出诊医师，号源的提供方与预约服务的履约方。核心操作包括个人信息维护、排班与号源查看、预约用户信息查询，是系统医生端工作台的主要使用者。
3.  **系统管理员角色**：系统最高权限管理者，负责系统全业务的配置、管理、监控与运维。核心操作包括科室与医生信息管理、排班与号源管理、用户管理、订单管理，是系统管理后台的主要使用者。

**2.4 系统功能模块总体划分**

基于业务流程与用户角色划分，本系统整体分为三大功能集群，对应三类用户的使用需求，总体模块划分如下：

1.  **患者端功能集群**：面向患者用户，核心包含用户注册与登录、科室与医生信息浏览、号源查询与在线预约、个人订单管理、个人中心五大核心模块。
2.  **医生端功能集群**：面向出诊医生，核心包含个人信息管理、排班与号源查看、预约用户管理三大核心模块。
3.  **管理后台功能集群**：面向系统管理员，核心包含系统基础管理、科室与医生管理、排班与号源管理、用户管理、订单管理、运营数据统计六大核心模块。

**2.5 业务约束与通用规则**

1.  **用户权限规则**：用户完成账号注册并成功登录后，即可解锁系统全部预约挂号相关功能，无额外身份核验门槛；未注册/未登录用户无法访问系统任何功能与页面。
2.  **号源管控规则**：号源总量由管理员在排班计划中设定，号源状态分为可预约、已约满、已下架三类，状态变更需同步更新缓存与数据库，保证全系统号源信息实时一致。
3.  **预约行为规则**：同一用户，同一医生同一出诊时段仅可预约1个号源，避免恶意占号行为，保障号源分配公平性。
4.  **预约取消规则**：用户可随时对已提交且未就诊的预约订单发起取消申请，无时间与次数限制；取消申请提交后，系统将即时释放对应号源，同步更新号源库存与订单状态。
5.  **合规性约束**：系统所有业务流程与数据管理需符合《中华人民共和国个人信息保护法》等相关法律法规中关于用户个人信息的基础管理要求。

**三、用户角色与权限需求**

**3.1 总体权限设计原则**

本系统采用基于角色的访问控制（RBAC）模型进行权限体系设计，核心遵循以下原则：

1.  **最小权限原则**：每个角色仅分配完成对应业务所需的最小功能与数据权限，杜绝权限过度分配与越权操作风险。
2.  **角色分离原则**：患者用户、医生、系统管理员三类角色业务边界完全隔离，对应独立的功能入口与操作界面，互不交叉。
3.  **数据隔离原则**：不同角色仅可访问权限范围内的数据，非授权用户无法跨角色、跨范围访问敏感数据，保障数据安全。

**3.2 患者用户角色权限**

**3.2.1 角色定位**

系统核心服务对象，为有就医挂号需求的普通用户，完成账号注册并登录后，即可使用预约挂号全流程功能，是患者端门户的唯一使用角色，支持自主注册账号。

**3.2.2 功能操作权限**

- 账号管理权限：支持账号注册、登录、密码修改、个人基础信息维护、账号安全退出操作。
- 信息浏览权限：支持查看医院科室列表、科室详情、出诊医生列表、医生个人简介、出诊排班计划、号源剩余情况等信息。
- 预约挂号权限：支持选择目标医生与出诊时段提交预约申请，完成号源锁定与预约订单生成操作。
- 订单管理权限：支持查看本人所有预约订单的详情、订单状态变更记录，可随时对未就诊的预约订单发起取消操作。

**3.2.3 数据访问权限**

- 仅可访问、编辑本人的账号信息与个人基础信息。
- 仅可查看本人提交的预约订单全量数据。
- 可访问医院科室、医生、出诊排班、号源库存的公开数据。
- 不可访问其他用户的账号信息、预约数据，不可访问医生、管理员的非公开业务数据。

**3.3 医生角色权限**

**3.3.1 角色定位**

医院出诊医师，号源的提供方与预约服务的履约方，是医生端工作台的唯一使用角色，账号由系统管理员统一创建分配，不支持自主注册。

**3.3.2 功能操作权限**

- 账号管理权限：支持账号登录、密码修改、个人执业信息、出诊简介维护、账号安全退出操作。
- 号源管理权限：支持查看本人所有排班对应的号源生成情况、号源剩余量、预约状态统计数据。
- 预约管理权限：支持查看本人出诊时段内的预约用户列表、预约订单详情，可核对预约用户的基础信息。

**3.3.3 数据访问权限**

- 仅可访问、编辑本人的账号信息、执业信息、排班计划数据。
- 仅可查看本人出诊排班对应的号源数据、预约订单数据、预约用户基础信息。
- 可查看本人所属科室的基础公开信息。
- 不可访问其他医生的排班、号源、预约数据，不可访问管理员后台的系统配置数据，不可访问非本人接诊的用户隐私数据。

**3.4 系统管理员角色权限**

**3.4.1 角色定位**

系统最高权限管理者，负责系统全业务的配置、管理、监控与运维，是系统管理后台的唯一使用角色，账号为系统预置，不支持自主注册。

**3.4.2 功能操作权限**

- 系统管理权限：支持系统基础参数配置、角色权限配置、功能菜单管理、系统运行状态监控操作。
- 科室与医生管理权限：支持医院科室信息的新增、编辑、删除、启停管理；支持医生账号的创建、分配、编辑、启停、所属科室配置管理。
- 排班与号源管理权限：支持为医生创建、编辑、删除排班计划，发布/下架排班号源、编辑号源规则、查看全系统号源数据、处理异常号源状态。
- 用户管理权限：支持查看全系统注册用户列表、用户基础信息、用户预约记录，可对异常行为账号进行启停管理。
- 订单管理权限：支持查看全系统所有预约订单的全量数据、订单状态追踪、异常订单处理、强制订单取消操作。
- 数据统计权限：支持查看科室挂号量统计、医生出诊数据统计、号源预约率统计、用户预约行为统计等全量运营数据。
- 账号安全退出权限：支持安全退出当前登录账号，清除会话信息。

**3.4.3 数据访问权限**

- 拥有系统全量业务数据的访问权限，包括用户数据、医生数据、科室数据、排班数据、号源数据、订单数据、系统配置数据。
- 可编辑系统内所有业务数据与配置数据，可对除预置超级管理员外的所有账号进行权限与状态管理。

**3.5 权限管控通用规则**

1.  所有角色必须通过账号密码登录验证后，方可访问对应权限内的功能与数据；未登录用户无法访问系统任何页面与功能。
2.  系统权限与角色直接绑定，用户仅可通过分配对应角色获得相关权限，不可越级获取非授权功能与数据访问权限。
3.  账号停用后，将立即失去系统登录权限，无法进行任何业务操作，历史数据仅可由管理员查看，不可被账号持有人访问。
4.  所有角色的密码均需采用加密方式存储，不可明文存储，杜绝账号信息泄露风险。

**四、功能性需求**

**4.1 总体功能概述**

本系统功能性需求基于三类核心用户角色的业务场景划分，整体分为**患者端功能集群、医生端功能集群、管理后台功能集群**三大模块，完整覆盖号源生成与发布、在线预约挂号、订单全生命周期管理、系统运维管控全业务流程。所有功能设计均遵循已确认的业务规则，保障业务流程闭环、操作逻辑顺畅，同时满足高并发预约场景下的业务正确性与稳定性要求，所有功能仅对已登录用户开放。

**4.2 患者端核心功能需求**

患者端为面向普通挂号用户的前端门户，**所有功能仅对完成注册并登录的用户开放，无未登录访问权限**，用户必须先完成账号注册并通过身份校验登录后，方可使用系统内所有功能。

**4.2.1 用户账号管理功能**

1.  **账号注册功能**功能描述：支持用户自主完成系统账号注册，为用户提供唯一的系统身份标识。
    1.  输入内容：用户名、登录密码、确认密码，无强制必填的额外信息项。
    2.  业务规则：用户名不可与系统内已注册账号重复；密码需满足基础安全规则；注册成功后账号即时生效，可直接登录系统。
2.  **账号登录功能**功能描述：完成用户身份合法性校验，校验通过后授予对应患者角色权限，进入系统门户。
    1.  输入内容：已注册的用户名、对应登录密码。
    2.  业务规则：用户名与密码匹配校验通过后方可登录；校验失败需给出明确提示，不泄露具体错误项；登录成功后建立用户会话，维持登录状态。
3.  **密码修改功能**功能描述：支持登录后的用户自主修改账号登录密码。
    1.  输入内容：原密码、新密码、确认新密码。
    2.  业务规则：原密码校验通过后方可修改；新密码需与原密码不一致；修改成功后需清除当前会话，用户需使用新密码重新登录。
4.  **个人信息维护功能**功能描述：支持登录后的用户自主编辑、维护个人基础信息。
    1.  可编辑内容：用户昵称、备注信息等非强制项，无必填的身份类信息。
    2.  业务规则：用户仅可编辑本人的个人信息，修改后即时生效。
5.  **账号安全退出功能**功能描述：支持用户安全退出当前登录账号，清除用户会话信息，退出后需重新登录方可使用系统功能。

**4.2.2 科室与医生信息浏览功能**

1.  **科室信息浏览功能**功能描述：支持登录后的用户查看医院全量科室列表，点击可进入科室详情页，查看科室介绍、所属出诊医生列表。
    1.  业务规则：科室列表支持按科室名称检索。
2.  **医生信息浏览功能**功能描述：支持登录后的用户查看对应科室下的出诊医生列表，点击可进入医生详情页，查看医生个人简介、执业信息、出诊排班计划。
    1.  业务规则：医生列表支持按医生姓名检索。
3.  **排班与号源查看功能**功能描述：支持登录后的用户查看目标医生的出诊排班计划，对应出诊时段的号源总量、剩余可预约数量、预约状态。
    1.  业务规则：号源状态实时更新，仅展示已发布的有效排班号源；已约满的时段需明确标识，不可提交预约申请。

**4.2.3 在线预约挂号功能**

- 功能描述：本功能为系统核心业务功能，支持登录后的用户选择目标医生、出诊时段提交预约申请，完成号源锁定与预约订单生成。
- 操作流程：用户选择可预约的医生出诊时段→提交预约申请→系统完成业务规则校验→校验通过后锁定并扣减对应号源→生成预约订单，完成预约流程。
- 输入内容：目标医生、出诊时段信息。
- 业务规则： 仅完成注册并登录的用户可提交预约申请。
    - 同一用户，同一医生的同一出诊时段仅可提交1个预约申请，不可重复预约。
    - 预约申请提交时，系统需实时校验号源剩余数量，仅当号源库存≥1时方可通过校验。
    - 高并发场景下，需严格保证号源扣减的准确性，杜绝号源超卖问题。
    - 预约成功后，号源状态同步更新，对应时段可预约数量即时扣减。

**4.2.4 预约订单管理功能**

1.  **订单列表查询功能**功能描述：支持登录后的用户查看本人提交的所有预约订单列表，可按订单状态（待就诊、已取消、已完成）筛选查看。
    1.  业务规则：仅展示当前登录用户本人的预约订单，不可查看其他用户的订单数据。
2.  **订单详情查看功能**功能描述：支持登录后的用户点击订单查看详情，包括预约医生信息、出诊时段、订单状态、提交时间等全量订单信息。
3.  **订单取消功能**功能描述：支持登录后的用户随时对未就诊的预约订单发起取消申请。
    1.  业务规则： 仅未就诊状态的订单可发起取消，无取消时间、取消次数限制。
        1.  取消申请提交后，系统即时释放对应号源，号源库存同步增加，可被其他用户预约。
        2.  订单取消后，订单状态同步更新为“已取消”，保留订单历史记录。

**4.2.5 个人中心功能**

- 功能描述：为登录后的用户提供个人信息、预约订单、账号设置的统一入口，整合个人信息维护、密码修改、订单查询、安全退出等功能，保障用户操作便捷性。

**4.3 医生端核心功能需求**

医生端为面向出诊医师的专属工作台，账号仅可由系统管理员统一创建分配，不支持用户自主注册，仅完成登录的医生可使用对应功能。

**4.3.1 账号与个人信息管理功能**

1.  **账号登录功能**功能描述：完成医生身份合法性校验，校验通过后授予对应医生角色权限，进入医生工作台。
    1.  输入内容：管理员分配的账号用户名、登录密码。
    2.  业务规则：用户名与密码匹配校验通过后方可登录；校验失败需给出明确提示；登录成功后建立用户会话，维持登录状态。
2.  **密码修改功能**功能描述：支持登录后的医生自主修改账号登录密码。
    1.  输入内容：原密码、新密码、确认新密码。
    2.  业务规则：原密码校验通过后方可修改；修改成功后需清除当前会话，需使用新密码重新登录。
3.  **个人信息维护功能**功能描述：支持登录后的医生编辑维护个人执业信息、出诊简介、专业擅长等公开信息，编辑后即时生效。
    1.  业务规则：医生仅可编辑本人的个人信息，不可修改账号所属科室、账号状态等核心配置项。
4.  **账号安全退出功能**功能描述：支持医生安全退出当前登录账号，清除会话信息，退出后需重新登录方可使用工作台功能。

**4.3.2 排班与号源查看功能**

- 功能描述：支持登录后的医生查看本人所有排班对应的号源数据，包括出诊日期、出诊时段、号源总量、已预约数量、剩余可预约数量、号源状态。
- 业务规则：仅可查看本人排班对应的号源数据，不可查看其他医生的号源信息；号源数据实时更新，支持按出诊日期筛选查看。

**4.3.3 预约用户管理功能**

1.  **预约列表查看功能**功能描述：支持登录后的医生查看本人出诊时段内的预约用户列表，可按出诊日期、出诊时段筛选查看。
    1.  业务规则：仅可查看本人出诊对应的预约记录，不可查看其他医生的预约信息。
2.  **预约详情查看功能**功能描述：支持登录后的医生查看单条预约订单的详情，包括预约用户基础信息、预约提交时间、订单状态等信息。

**4.4 管理后台核心功能需求**

管理后台为面向系统管理员的运维管控平台，账号为系统预置，不支持自主注册，仅完成登录的管理员可使用对应功能，拥有系统全量业务的管控权限。

**4.4.1 系统基础管理功能**

1.  **系统参数配置功能**功能描述：支持登录后的管理员配置系统基础运行参数，包括号源发布规则、预约业务基础规则、页面展示配置等。
2.  **角色权限管理功能**功能描述：支持登录后的管理员对系统内三类核心角色的功能权限、数据权限进行配置管理，遵循最小权限原则。
3.  **功能菜单管理功能**功能描述：支持登录后的管理员对系统前端功能菜单进行新增、编辑、排序、启停管理，适配权限配置需求。
4.  **系统运行状态监控功能**功能描述：支持登录后的管理员查看系统核心服务运行状态、接口响应状态、数据库连接状态，及时发现系统运行异常。
5.  **账号安全退出功能**功能描述：支持登录后的管理员安全退出当前登录账号，清除会话信息，退出后需重新登录方可访问管理后台。

**4.4.2 科室信息管理功能**

- 功能描述：支持登录后的管理员对医院科室基础信息进行全生命周期管理，为医生排班、用户挂号提供基础数据支撑。
- 核心子功能：科室信息新增、科室信息编辑、科室启停管理、科室删除管理。
- 业务规则：科室名称不可重复；已停用的科室不可新增医生、不可发布新的排班号源；已关联医生与排班数据的科室不可直接删除。

**4.4.3 医生账号与信息管理功能**

- 功能描述：支持登录后的管理员对医生账号进行全生命周期管理，是医生账号的唯一创建入口。
- 核心子功能：医生账号创建、账号信息编辑、账号启停管理、医生所属科室配置。
- 业务规则：医生账号用户名不可重复；账号停用后，医生将无法登录系统，已发布的有效排班号源不受影响；医生编辑后的个人信息即时生效，无需审核。

**4.4.4 排班与号源管理功能**

- 功能描述：本功能为管理后台核心业务功能，支持登录后的管理员直接为医生创建、编辑、删除出诊排班计划，完成号源的生成、发布、下架全流程管理，无需医生申请。
- 核心子功能： **排班创建功能**：支持管理员为指定医生创建出诊排班，可设置出诊日期、出诊时段、单时段号源总量，排班创建后系统自动生成对应号源。
    - **排班编辑功能**：支持管理员对所有排班计划（含未发布、已发布状态）进行内容编辑修改。
    - **排班删除功能**：支持管理员删除未发布的排班计划，已发布的排班不可直接删除，需先下架号源。
    - **号源发布功能**：管理员可对已创建的排班号源执行发布操作，发布后的号源对外展示，可被用户预约。
    - **号源下架功能**：支持管理员对已发布的号源执行下架操作，下架后的号源不可被用户预约，已完成的预约订单不受影响。
    - **全量号源管理功能**：支持管理员查看系统全量号源数据，可按科室、医生、出诊日期筛选查看，处理号源状态异常问题。
- 业务规则： 同一医生同一时段仅可创建一条排班计划，不可重复设置。
    - 排班计划需明确号源总量，号源总量需为正整数。
    - 号源状态变更需同步更新缓存与数据库，保证全系统数据一致性。

**4.4.5 注册用户管理功能**

- 功能描述：支持登录后的管理员查看系统全量注册用户列表，可查看用户基础信息、历史预约记录，可对异常行为账号执行启停管理。
- 业务规则：账号停用后，用户将无法登录系统，不可提交新的预约申请，已完成的有效预约订单不受影响。

**4.4.6 预约订单全量管理功能**

- 功能描述：支持登录后的管理员查看系统全量预约订单数据，完成订单全流程管控与异常处理。
- 核心子功能：全量订单列表查询、订单详情查看、异常订单处理、强制订单取消操作。
- 业务规则：管理员可对异常订单执行强制取消操作，取消后同步释放对应号源，更新订单状态；所有订单操作需严格遵循业务规则，保障数据一致性。

**4.4.7 运营数据统计功能**

- 功能描述：支持登录后的管理员查看系统运营核心统计数据，为医院运营管理提供数据支撑。
- 核心统计维度：科室挂号量统计、医生出诊数据统计、号源预约率统计、用户预约行为统计。
- 业务规则：统计数据可按日期范围筛选，数据需与业务原始数据保持一致，保证统计准确性。

**五、非功能性需求**

本章节非功能性需求围绕本科毕业设计的开发、测试、论文撰写与答辩演示场景设计，以保障系统核心功能正常实现、业务逻辑可验证、演示流程顺畅为核心目标，不设置商用级别的严苛指标与冗余防护要求，具体内容如下：

**5.1 性能需求**

1.  **页面响应性能**：系统静态页面加载完成时间≤2s，带动态数据渲染的业务页面加载完成时间≤3s，无白屏、卡顿、样式错乱问题，满足答辩演示的基础流畅度要求。
2.  **接口响应性能**：常规信息查询类接口平均响应时间≤800ms；预约提交、订单取消、排班编辑等核心业务接口平均响应时间≤1.5s，无明显请求延迟。
3.  **并发处理性能**：核心预约抢号场景，支持30个并发用户同时提交预约申请，系统无崩溃、无请求丢失，可保证号源扣减准确、无超卖，满足毕设并发测试的基础验证要求；常规业务场景支持100个用户同时在线操作，功能可正常运行。
4.  **数据同步性能**：号源状态、订单状态、排班信息发生变更后，系统内各页面数据同步更新延迟≤1s，无数据展示不一致的问题。

**5.2 安全性需求**

本系统安全性需求以保障基础业务权限隔离、账号安全、核心数据完整为目标，不设置商用级复杂攻击防护措施，具体要求如下：

1.  **登录权限控制**：系统所有功能仅对已登录用户开放，无未登录访问入口，未登录用户无法访问任何业务页面与接口，所有业务请求需携带合法的用户身份凭证。
2.  **角色权限管理**：采用RBAC基础角色权限模型，患者、医生、管理员三类角色的功能与数据边界清晰，仅可访问自身权限范围内的内容，无越权操作漏洞。
3.  **账号密码安全**：用户密码采用不可逆加密方式存储，禁止明文存储；密码修改需校验原密码的合法性，保障账号基础安全。
4.  **核心数据完整**：预约订单、号源库存、排班信息等核心业务数据，需保证操作前后的完整性与一致性，不会因正常业务操作出现数据篡改、丢失的问题。

**5.3 易用性需求**

1.  **操作流程简洁**：核心预约挂号、排班管理等高频业务操作步骤不超过3步，无冗余跳转与操作，符合常规用户的使用习惯。
2.  **界面逻辑清晰**：系统界面布局规整，导航明确，不同角色的功能入口区分清晰，无混淆操作；功能命名通俗易懂，无专业理解壁垒。
3.  **提示信息明确**：用户操作成功、失败、受业务规则限制时，均给出清晰易懂的提示，明确告知操作结果与问题原因，无模糊、笼统的报错内容。
4.  **操作容错友好**：对排班删除、订单强制取消等关键操作设置二次确认，避免误操作；表单输入给出基础格式提示，减少用户操作错误。

**5.4 兼容性需求**

1.  **浏览器兼容性**：系统前端页面兼容Chrome、Edge等主流桌面浏览器的最新正式版本，在目标浏览器中无样式错乱、功能失效、交互异常的问题，满足答辩演示的环境要求。
2.  **分辨率适配**：系统界面自适应1920×1080、1366×768等主流桌面设备分辨率，不同分辨率下无内容遮挡、排版错乱问题。
3.  **运行环境兼容**：后端服务兼容Windows、Linux主流操作系统，适配JDK 8、MySQL 5.7/8.0、Redis 5.0及以上常规稳定版本，满足本地开发、部署与演示的需求。

**5.5 可靠性需求**

1.  **功能可用性**：在毕业设计开发、测试与答辩演示期间，系统核心业务功能（预约挂号、排班管理、订单管理等）可用率100%，无功能失效、服务宕机问题。
2.  **数据可靠性**：系统所有业务数据均持久化存储至数据库，无数据丢失风险；号源扣减、订单生成等核心操作保证事务一致性，不会出现部分成功、部分失败的异常情况。
3.  **异常容错性**：系统对常规的输入错误、操作异常具备基础的容错处理能力，不会因单个用户的错误操作导致系统崩溃、服务不可用。

**5.6 可维护性需求**

1.  **架构规范**：系统采用前后端分离的分层架构，代码模块低耦合、高内聚，符合软件工程开发规范，便于功能修改与调试。
2.  **代码可读性**：核心业务代码配有清晰的注释，命名规范统一，逻辑结构清晰，便于论文撰写过程中的功能讲解与后续优化调整。
3.  **部署可复现性**：系统配套完整的开发、部署说明文档，明确环境配置、启动流程、功能模块说明，保障系统可复现、可维护。

**六、接口需求**

本章节接口需求围绕系统前后端分离架构设计，贴合毕业设计开发、联调、测试与论文撰写需求，以保障核心业务数据交互顺畅、接口逻辑可验证为核心目标，不设置商用级冗余校验规则，接口设计风格统一、语义清晰，便于开发实现与功能测试。

**6.1 接口总体设计规范**

1.  **架构适配**：本系统采用前后端分离架构，前端Vue应用与后端Spring Boot服务之间，通过标准化的HTTP接口完成所有业务数据交互，无服务端页面渲染逻辑。
2.  **风格规范**：接口统一采用RESTful设计风格，通过GET、POST、PUT、DELETE四种标准请求方式，分别对应数据的查询、新增、修改、删除操作，保证接口语义清晰、风格统一。
3.  **数据格式**：所有接口的请求体与响应体均采用JSON格式，编码格式统一为UTF-8，无特殊格式要求，适配前后端开发与联调需求。
4.  **响应规范**：所有接口采用统一的响应数据结构，包含状态码、响应提示、业务数据三个核心字段，便于前端统一处理请求结果与异常提示。

**6.2 核心业务接口需求**

**6.2.1 用户账号管理接口**

- 账号注册接口：实现用户账号的注册功能，请求方式POST，无登录权限要求，完成用户名唯一性校验、密码格式校验，校验通过后完成账号数据入库。
- 账号登录接口：实现用户身份校验与登录功能，请求方式POST，无登录权限要求，完成用户名与密码的合法性校验，校验通过后返回用户身份令牌与角色信息，建立用户会话。
- 密码修改接口：实现登录用户的密码修改功能，请求方式PUT，权限要求：所有已登录用户，完成原密码校验后更新密码数据。
- 个人信息维护接口：实现登录用户的个人基础信息查询与编辑功能，请求方式GET（查询）、PUT（编辑），权限要求：所有已登录用户，仅可操作本人的信息数据。

**6.2.2 科室与医生信息接口**

- 科室列表查询接口：实现医院全量科室信息的查询功能，请求方式GET，权限要求：所有已登录用户。
- 科室信息管理接口：实现科室信息的新增、编辑、删除、启停功能，请求方式POST（新增）、PUT（编辑/启停）、DELETE（删除），权限要求：仅系统管理员可操作。
- 医生列表查询接口：实现对应科室下的医生信息、出诊排班信息查询功能，请求方式GET，权限要求：所有已登录用户。
- 医生账号管理接口：实现医生账号的创建、编辑、启停、所属科室配置功能，请求方式POST（创建）、PUT（编辑/启停），权限要求：仅系统管理员可操作。
- 医生个人信息查询接口：实现登录医生的个人信息查询与编辑功能，请求方式GET（查询）、PUT（编辑），权限要求：仅医生角色可操作本人信息。

**6.2.3 排班与号源管理接口**

- 排班创建接口：实现管理员为指定医生创建出诊排班计划的功能，请求方式POST，权限要求：仅系统管理员可操作，创建后自动生成对应号源数据。
- 排班编辑接口：实现管理员对所有排班计划（含未发布、已发布状态）的编辑修改功能，请求方式PUT，权限要求：仅系统管理员可操作。
- 排班删除接口：实现管理员对未发布排班计划的删除功能，请求方式DELETE，权限要求：仅系统管理员可操作。
- 号源发布与下架接口：实现管理员对排班号源的发布、下架操作，请求方式PUT，权限要求：仅系统管理员可操作。
- 号源数据查询接口：实现号源总量、剩余量、状态信息的查询功能，请求方式GET，权限要求：所有已登录用户，医生仅可查询本人排班的号源数据。
- 医生个人排班查询接口：实现登录医生的个人排班计划、号源数据查询功能，请求方式GET，权限要求：仅医生角色可操作本人数据。

**6.2.4 预约挂号与订单管理接口**

- 预约提交接口：实现用户预约挂号申请的提交功能，请求方式POST，权限要求：仅患者角色可操作，完成号源库存校验、用户预约资格校验，校验通过后完成号源扣减与预约订单生成。
- 个人订单列表查询接口：实现登录用户的本人预约订单列表查询功能，请求方式GET，权限要求：仅患者角色可操作本人订单数据。
- 订单详情查询接口：实现预约订单全量信息的查询功能，请求方式GET，权限要求：患者仅可查询本人订单，医生仅可查询本人出诊对应的订单，管理员可查询全量订单。
- 订单取消接口：实现用户对未就诊预约订单的取消功能，请求方式PUT，权限要求：仅患者角色可操作本人订单，取消后即时释放对应号源，更新订单状态。
- 全量订单管理接口：实现管理员对全系统预约订单的查询、异常处理、强制取消功能，请求方式GET（查询）、PUT（强制取消），权限要求：仅系统管理员可操作。
- 医生预约订单查询接口：实现登录医生的本人出诊时段预约订单、用户信息查询功能，请求方式GET，权限要求：仅医生角色可操作本人对应数据。

**6.3 接口通用要求**

1.  **身份校验要求**：除账号注册、账号登录接口外，其余所有业务接口必须携带合法的用户身份令牌，接口接收到请求后需先完成身份与权限校验，校验不通过的请求直接拦截并返回错误提示，符合系统全功能需登录使用的规则。
2.  **参数校验要求**：所有接口需对请求参数进行基础合法性校验，对必填参数缺失、参数格式错误的请求，直接返回明确的错误提示，不执行核心业务逻辑，避免因参数异常导致系统错误。
3.  **异常处理要求**：所有接口需具备基础的异常处理能力，对业务执行过程中的异常情况，返回清晰的错误提示信息，不直接抛出系统底层异常，保障接口调用的稳定性。
4.  **可测试性要求**：所有接口支持通过Postman等接口测试工具进行独立测试，接口功能与业务逻辑可单独验证，适配毕业设计的接口测试、集成测试需求。

2.  核心业务数据需实现全流程持久化存储，保证数据的完整性、可追溯性，满足业务流程闭环、功能测试、答辩演示的全场景需求。
3.  数据设计需与已确认的业务规则、功能需求完全匹配，无超出业务范围的冗余数据项，同时预留基础的扩展字段，适配毕业设计后期的功能优化需求。

**7.2 核心数据实体与核心属性**

本系统核心数据实体及对应核心属性如下，所有实体均设置唯一主键作为数据唯一标识，保证数据关联性与唯一性：

1.  **患者用户信息实体**
2.  核心属性：用户ID（主键）、用户名、加密存储的登录密码、用户昵称、账号状态（正常/停用）、账号创建时间、信息更新时间。**管理员信息实体**
3.  核心属性：管理员ID（主键）、用户名、加密存储的登录密码、账号创建时间、信息更新时间。**科室信息实体**
4.  核心属性：科室ID（主键）、科室名称、科室简介、科室状态（启用/停用）、创建时间、更新时间。**医生信息实体**
5.  核心属性：医生ID（主键）、关联登录账号ID、医生姓名、所属科室ID、执业简介、专业擅长、账号状态（正常/停用）、创建时间、更新时间。**排班计划实体**
6.  核心属性：排班ID（主键）、所属医生ID、所属科室ID、出诊日期、出诊时段、号源总量、排班状态（未发布/已发布/已下架）、创建时间、更新时间。**号源信息实体**
7.  核心属性：号源ID（主键）、关联排班ID、所属医生ID、号源总量、剩余可预约数量、号源状态（可预约/已约满/已下架）、创建时间、更新时间。**预约订单实体**

**核心属性：订单ID（主键）、关联预约用户ID、关联医生ID、关联排班ID、关联号源ID、预约科室名称、预约出诊日期、预约出诊时段、订单状态（待就诊/已取消/已完成）、订单创建时间、状态更新时间。7.3 数据存储需求**

1.  **核心存储介质**：系统全量业务结构化数据采用MySQL关系型数据库进行持久化存储；号源库存、高频访问的排班与科室数据采用Redis进行缓存存储，提升数据访问效率，同时适配并发预约场景的测试需求。
2.  **数据备份需求**：支持数据库手动备份与一键恢复功能，保障毕业设计开发、测试过程中数据不丢失，满足答辩演示环境的数据快速还原需求。
3.  **数据存储周期**：系统所有业务操作数据全程持久化保留，无自动清理规则，完整留存全业务流程的操作数据，便于论文撰写过程中的功能验证、数据展示与逻辑说明。

**7.4 数据一致性需求**

1.  **事务一致性**：号源扣减与预约订单生成、订单取消与号源释放等核心联动业务操作，需保证数据库事务的原子性，操作要么全部执行成功，要么全部回滚，杜绝部分成功、部分失败导致的数据不一致问题。
2.  **缓存与数据库一致性**：号源库存、排班状态等缓存数据，需与数据库中的核心数据实时同步，数据发生变更后即时更新缓存内容，保证全系统各页面展示的数据一致，无缓存与数据库数据偏差问题。
3.  **业务数据一致性**：已发布的排班计划编辑修改后，已生成的有效预约订单核心信息保持不变；号源总量变更后，需同步更新剩余可预约数量，保障业务数据的连续性与一致性。
4.  **权限数据一致性**：用户角色、账号状态、权限配置发生变更后即时生效，保证权限管控与数据访问范围的一致性，无权限滞后、越权访问的问题。
