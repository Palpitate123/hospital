# 一、文档概述

## 1.1 文档目的

本文档完整定义医院预约挂号系统的数据库设计方案，核心覆盖**ER实体关系、表结构详细设计、表关联关系、全量数据字典**四大核心模块，是系统开发、测试、部署、论文撰写的核心依据，严格贴合毕业设计的业务需求与研究目标，重点保障高并发预约场景下的数据一致性、业务规则合规性与数据安全性。

## 1.2 文档范围

本文档覆盖系统全业务流程的数据库设计，包含**系统权限、医院基础信息、排班号源管理、预约订单管理**四大核心业务模块，完整适配患者、医生、系统管理员三类用户的全量业务需求，无商用级冗余设计，完全适配毕业设计开发与演示场景。

## 1.3 核心设计规范

|     |     |
| --- | --- |
| **规范类型** | **详细规则** |
| 命名规范 | 1\. 表名、字段名统一使用小写英文字母+下划线命名，禁止使用SQL关键字、特殊字符；<br><br>2\. 系统通用表以sys_为前缀，业务表以hos_为前缀；<br><br>3\. 主键统一命名为id，外键统一命名为关联表名_id，如user_id、dept_id |
| 数据类型规范 | 1\. 主键统一使用bigint自增类型，避免使用UUID；<br><br>2\. 状态、类型等枚举字段使用tinyint类型，节省存储空间；<br><br>3\. 字符串字段统一使用varchar类型，非必要不使用text类型；<br><br>4\. 时间字段统一使用datetime类型，不使用timestamp |
| 约束规范 | 1\. 所有表必须设置主键，保证记录唯一性；<br><br>2\. 业务层面要求唯一的字段/字段组合，必须设置唯一约束；<br><br>3\. 非空字段必须设置NOT NULL约束，核心字段设置默认值；<br><br>4\. 号源库存等数值字段设置非负约束，从底层杜绝超卖问题 |
| 索引规范 | 1\. 主键默认创建主键索引，外键字段必须创建普通索引；<br><br>2\. 高频查询、筛选、排序字段创建普通索引；<br><br>3\. 联合索引遵循最左匹配原则，筛选度高的字段前置；<br><br>4\. 避免创建冗余索引、低基数字段单独索引 |

**二、ER实体关系图**

**2.1 核心实体定义**

本系统共定义8个核心数据实体，完整覆盖全业务流程，实体定义如下：

1.  **角色实体**：定义系统三类核心角色（系统管理员、医生、患者），对应权限边界
2.  **用户实体**：系统所有使用者的通用账号信息，是权限管控的核心主体
3.  **科室实体**：医院科室基础信息，是医生、排班的归属主体
4.  **医生实体**：出诊医生的执业信息，关联用户账号与所属科室
5.  **排班计划实体**：医生出诊的排班规则，是号源生成的核心依据
6.  **号源信息实体**：对应排班的可预约名额数据，是预约业务的核心库存主体
7.  **预约订单实体**：患者预约挂号的订单数据，记录预约全流程信息
8.  **操作日志实体**：用户核心操作审计记录，用于安全追溯与问题排查

| 排班计划实体 | 号源信息实体 | 一对一 | 一条排班计划对应一条号源数据，号源随排班生成 |
| 号源信息实体 | 预约订单实体 | 一对多 | 一个号源可生成多条预约订单（不超过号源总量），一条订单仅对应一个号源 |
| 用户实体 | 预约订单实体 | 一对多 | 一个患者可提交多条预约订单，一条订单仅归属一个患者 |

**六、核心业务约束说明**

1.  **防超卖约束**：号源表remain_number字段设置非负约束，配合悲观锁行级锁+乐观锁版本号，从数据库底层彻底杜绝号源超卖问题，贴合开题报告核心研究目标。
2.  **预约唯一性约束**：订单表设置联合唯一索引，保证同一用户同一医生同一时段仅可预约1次，避免恶意占号，符合需求规格说明书的业务规则。
3.  **排班唯一性约束**：排班表设置联合唯一索引，保证同一医生同一时段仅可创建一条排班，避免号源冲突。
4.  **事务一致性约束**：预约挂号、订单取消核心业务采用数据库事务，保证号源扣减/释放与订单状态变更的原子性，杜绝数据不一致问题。
5.  **权限隔离约束**：通过外键关联与业务查询条件，实现医生仅可查看本人数据、患者仅可查看本人订单，符合RBAC权限模型的最小权限原则。
