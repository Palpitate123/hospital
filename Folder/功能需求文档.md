## 一、项目核心需求提炼

### 1.1 系统目标

本系统是面向中小型医疗机构的轻量化、高可用的线上预约挂号服务平台，采用 **Spring Boot + Vue.js** 前后端分离架构，核心目标如下：

| 目标维度 | 具体内容 |
|---------|---------|
| **业务目标** | 构建全线上化的预约挂号服务体系，替代传统线下挂号模式，解决排队耗时长、号源信息不透明、资源分配不均等痛点 |
| **技术目标** | 攻克高并发预约场景下的数据一致性与业务公平性问题，杜绝号源超卖，保障系统稳定性 |
| **安全目标** | 构建基础的用户数据安全防护体系，实现身份认证、权限管控、数据隔离 |
| **应用目标** | 为毕业设计提供完整的开发、测试实践案例，形成可复用的技术参考 |

### 1.2 用户角色需求

系统定义三类核心用户角色，各角色需求如下：

#### 1.2.1 患者用户（核心服务对象）

| 需求类型 | 具体需求 |
|---------|---------|
| 账号需求 | 支持自主注册账号、登录、密码修改、个人信息维护 |
| 信息需求 | 查看科室列表、医生信息、出诊排班、号源剩余情况 |
| 预约需求 | 选择目标医生与时段提交预约申请，完成号源锁定与订单生成 |
| 订单需求 | 查看本人预约订单、取消未就诊订单 |

#### 1.2.2 医生用户（服务履约方）

| 需求类型 | 具体需求 |
|---------|---------|
| 账号需求 | 使用管理员分配的账号登录、修改密码、维护个人执业信息 |
| 排班需求 | 查看本人排班计划及对应号源数据（总量、剩余量、状态） |
| 预约需求 | 查看本人出诊时段的预约用户列表及订单详情 |

#### 1.2.3 系统管理员（运维管控方）

| 需求类型 | 具体需求 |
|---------|---------|
| 基础管理 | 系统参数配置、角色权限管理、功能菜单管理、运行状态监控 |
| 科室管理 | 科室信息的新增、编辑、删除、启停管理 |
| 医生管理 | 医生账号创建、编辑、启停、所属科室配置 |
| 排班管理 | 为医生创建/编辑/删除排班计划，发布/下架号源 |
| 用户管理 | 查看注册用户列表，对异常账号执行启停管理 |
| 订单管理 | 查看全量订单数据，处理异常订单，强制取消订单 |
| 数据统计 | 科室挂号量、医生出诊数据、号源预约率等统计 |

### 1.3 核心业务流程

系统核心业务围绕 **号源生命周期** 与 **预约挂号全流程** 展开：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        号源生成与发布环节                            │
│  管理员配置科室/医生 → 创建排班计划 → 设置号源总量 → 发布号源        │
└─────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────┐
│                        用户预约挂号环节                              │
│  用户注册登录 → 浏览科室/医生 → 选择时段 → 提交预约 → 生成订单       │
└─────────────────────────────────────────────────────────────────────┘
                                    ↓
┌─────────────────────────────────────────────────────────────────────┐
│                        订单履约与管理环节                            │
│  用户查看订单 → 按时就诊/取消订单 → 医生查看预约用户 → 状态更新      │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.4 核心业务规则

| 规则类型 | 规则内容 |
|---------|---------|
| **权限规则** | 所有功能仅对已登录用户开放，未登录用户无法访问任何页面 |
| **预约唯一性** | 同一用户、同一医生、同一出诊时段仅可预约1次 |
| **号源管控** | 号源总量由管理员设定，状态分为：可预约、已约满、已下架 |
| **取消规则** | 用户可随时取消未就诊订单，取消后即时释放号源 |
| **防超卖** | 通过非负约束 + 乐观锁 + 悲观锁保障号源扣减准确性 |

| M3.2 | 预约申请提交 | 提交预约申请，完成号源锁定 | 医生ID、排班ID、号源ID | 预约结果、订单信息 | 1. 同一用户同一医生同一时段仅可预约1次<br>2. 实时校验号源库存≥1<br>3. 高并发下保证号源扣减准确 |
| M3.3 | 预约结果确认 | 展示预约成功信息 | 订单ID | 订单详情 | 预约成功后号源状态同步更新 |

**预约业务流程图：**

```
用户选择号源 → 提交预约申请
                    ↓
            ┌───────────────┐
            │ 用户登录校验   │ → 未登录 → 提示登录
            └───────────────┘
                    ↓ 已登录
            ┌───────────────┐
            │ 预约资格校验   │ → 已预约过 → 提示"不可重复预约"
            └───────────────┘
                    ↓ 合格
            ┌───────────────┐
            │ 号源库存校验   │ → 库存不足 → 提示"号源已满"
            └───────────────┘
                    ↓ 有库存
            ┌───────────────┐
            │ 号源锁定扣减   │ → 使用乐观锁+悲观锁
            └───────────────┘
                    ↓ 成功
            ┌───────────────┐
            │ 生成预约订单   │
            └───────────────┘
                    ↓
            ┌───────────────┐
            │ 更新号源状态   │
            └───────────────┘
                    ↓
              预约成功
```

| 管理员下架号源 | 号源不可预约 | source_status: 1→3 |

### 7.2 关键业务约束汇总

| 约束类型 | 约束内容 | 实现方式 |
|---------|---------|---------|
| 用户名唯一性 | 用户名不可重复 | 数据库唯一索引 |
| 科室名称唯一性 | 科室名称不可重复 | 数据库唯一索引 |
| 排班唯一性 | 同一医生同一时段仅一条排班 | 联合唯一索引(doctor_id, work_date, work_time) |
| 预约唯一性 | 同一用户同一医生同一时段仅预约1次 | 联合唯一索引(user_id, doctor_id, schedule_id) |
| 号源非负约束 | 剩余号源数≥0 | 数据库非负约束 + 乐观锁 |
| 防超卖 | 号源扣减准确性 | 乐观锁(version) + 悲观锁(行锁) + 事务 |
